● From the Rust book — a trait with a supertrait bound:

  use std::fmt;

  trait Printable: fmt::Display {
      fn print(&self) {

      }
  }

  And with multiple bounds using +:

  trait Storable: Clone + fmt::Debug {
      fn store(&self);
  }

  Any type implementing Storable must also implement Clone and Debug. The bounds are enforced at the impl site — if you try to impl Storable
  for Foo and Foo doesn't implement Clone, it won't compile.

❯ why tryfrom and not just from?                                                                                                              

● From guarantees the conversion always succeeds. TryFrom returns a Result to handle the case where it might fail.

  For Value → T, the conversion can fail — if you have a Value::Float and try to convert it to bool, that's an error. So TryFrom is the right
  choice.

  Going the other way (T → Value) always succeeds — every f32 is a valid Value::Float — so Into<Value> (equivalently From<Value> for T) is
  appropriate there.

❯ ok i get the functional difference, but it seems like we ought to be able to force the conversion to always work? or is catching the error

● The TryFrom failure is exactly the right place to catch config-layer mistakes — but you're right that it should be unreachable in a
  correctly constructed system.

  The key insight is: who calls TryFrom?

  Not the execute loop. The execute loop only ever calls bus.read::<f32>(&some_float_reader), and because SignalReader<f32> can only exist if
  a f32 was written there, the conversion is guaranteed to succeed. The unreachable! in the architecture doc reflects this — it's a logical
  impossibility, not a runtime risk.

  TryFrom only gets called during construction, when the config layer is wiring things up. That's where a type mismatch would be caught —
  startup time, not runtime. Which is exactly your constraint.

  So the picture is:
  - Compile time: typed handles (SignalReader<f32>) prevent wiring mismatches in code-defined graphs
  - Startup time: TryFrom catches mismatches in config-defined graphs — system fails to construct, never enters the run loop
  - Run loop: conversion is structurally guaranteed, unreachable! is genuinely unreachable

  No type errors can survive into the run loop. The TryFrom failure mode is the startup-time safety valve, not a runtime risk.

  On the duplication question — yes, the idiomatic Rust answer is a macro. Specifically a macro_rules! declarative macro. The pattern of

❯ tell me more about the heap issue? if I had blocks that inputed/outputed string signals, is that gonna be an issue?                         

● Yes, potentially. The issue is allocation in the execute loop, which we specifically ruled out as a constraint.

  Every time a block writes a String signal, it creates a new heap allocation. Every time the value gets cloned (e.g. multiple readers),
  another allocation. On a microcontroller with limited heap and no allocator guarantees, this adds up — and it's non-deterministic in timing,
   which matters for any kind of real-time behaviour.

  There are a few options:

  - Don't allow String as a signal type — strings are fine for config/serialization (loading a JSON config, naming signals) but not as runtime
   wire types. This is probably the right call given the use case.
  - Fixed-size string — something like [u8; N] with a const generic size baked in at compile time. Stack allocated, deterministic, but awkward
   to work with.
  - &'static str — string literals only, no allocation. Very limited.

  My instinct: keep strings out of the signal type system entirely. The config layer deals in strings (JSON keys, block names, signal names),
  but by the time you're in the execute loop everything has been resolved to numeric/boolean values. String signals don't fit any of the
  embedded use cases you've described.

  What would a string signal actually represent in your button box / hardware application?
